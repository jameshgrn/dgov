## Audit: DAG Runner & CLI Plumbing

### Critical Bugs
- **src/dgov/dag.py:900-910** — `run_dag` skip handling calls `close_worker_pane` with the task slug instead of the pane slug, so already-dispatched panes for newly skipped tasks are never actually closed. **Severity: P0.** **Suggested fix:** look up the pane slug for each skipped task from persistence (or track a task→pane mapping) and pass the correct pane slug into `close_worker_pane`, and add a regression test that verifies panes are closed when tasks are skipped.
- **src/dgov/batch.py:311-323** — In `run_batch`, timeout handling treats pane slugs as task IDs: it adds the pane slug to `failed_ids` and then feeds that set into `_transitive_dependents`, which expects task IDs. This corrupts the dependency graph and can skip the wrong tasks or fail to skip dependents of a truly failed task. **Severity: P0.** **Suggested fix:** keep a mapping from pane slug to task id (e.g. `slug_to_id`) and always add the underlying task id to `failed_ids` / `skipped_ids` and to the per-tier `tasks` status, never the pane slug.

### Logic Errors
- **src/dgov/batch.py:327-342** — Merge failure handling mirrors the timeout bug: it appends the pane slug to `failed_ids`/`results["failed"]` and feeds it into `_transitive_dependents`, again mixing pane slugs with task ids. **Severity: P1.** **Suggested fix:** use the same `slug_to_id` mapping as above during merge, so failures and skips are tracked per batch task id.
- **src/dgov/dag.py:959-962** — Partial execution path for `tier_limit` assumes tasks start in a `"pending"` state, but new runs initialize `task_states` as `{}` and the code never sets `"pending"`. As a result, `unexecuted` is always empty and a tier-limited run is reported as fully completed instead of partial. **Severity: P1.** **Suggested fix:** initialize all tasks to `"pending"` when creating a new run, or compute `unexecuted` as tasks not present in `task_states` and not skipped/merged.

### Inefficiencies
- **src/dgov/dag.py:874-891** — `run_dag` parses the DAG file and recomputes tiers twice: once before `_start_or_resume_run` and again inside `_start_or_resume_run` (which also calls `compute_tiers` for state_json). For large DAGs this doubles parsing and graph work per run. **Severity: P2.** **Suggested fix:** call `_start_or_resume_run` first, then reuse its `DagDefinition` and cached tiers/topological order instead of reparsing and recomputing.
- **src/dgov/dag.py:893-912** — Skip handling recomputes the list of `"dispatched"` tasks inside the loop for every skipped slug, and then attempts to close panes using the task slug (which also fails, see Critical Bugs). **Severity: P2.** **Suggested fix:** build the dispatched-set once, resolve each to its pane slug, close each pane at most once, and avoid the nested per-skip scan.

### Dead Code
- **src/dgov/dag.py:1126-1229** — `_retry_same_agent` and `_escalate_to_next_agent` are no longer referenced; `run_task_until_terminal` implements its own retry/escalation path instead. **Severity: P3.** **Suggested fix:** remove these helpers (and associated imports) or wire them into the orchestration path if they are still desired, to avoid confusion about which retry/escalation flow is authoritative.

### Minor Issues
- **src/dgov/tmux.py:14-24** — `_run` always raises `RuntimeError` with the raw tmux stderr and no context about the higher-level operation, and several callers treat all `RuntimeError`s equivalently. This makes diagnosing tmux misconfiguration vs. transient failures harder, and certain callers (e.g. `setup_governor_workspace`) silently ignore errors. **Severity: P3.** **Suggested fix:** wrap tmux failures with a more structured error (e.g. include the high-level operation and arguments) and consider logging at warning/error level in silent paths so tmux problems are actually visible.
- **src/dgov/cli/pane.py:37-48** — `pane_util` accepts an arbitrary `command` string and passes it straight to `create_utility_pane`, which sends it into a shell; this is expected for a power-user CLI, but the help text does not warn that the argument is executed as a shell command in tmux. **Severity: P3.** **Suggested fix:** explicitly document in the help/README that `pane util` runs the provided command string in a shell, and recommend quoting/escaping as with normal shell usage.
