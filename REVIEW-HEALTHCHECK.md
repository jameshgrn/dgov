# Health-check Adversarial Audit

## 1. Per-agent failure matrix

| Agent / family | What breaks | When / assumption | Detection and latency |
| --- | --- | --- | --- |
| `pi` (user config in `~/.dgov/agents.toml:9`) | Health probe is wired to `curl -sf http://localhost:8080/health`, but the River server actually lives on a remote GPU; the check therefore always fails even when the CLI binary exists. | `run_preflight` invokes `check_agent_health` with a 10 s timeout (`src/dgov/preflight.py:363`), and `create_worker_pane` runs the same command without any timeout before the agent launch (`src/dgov/lifecycle.py:441`). | `check_agent_health` returns failure inside ~10 s, so `dgov preflight` / `dgov worker` abort before a pane is created. If somehow bypassed, the unbounded `curl` in `create_worker_pane` hangs forever and raises `RuntimeError` only after the command fails. |
| API-style built-ins (`claude`, `codex`, `gemini`, `opencode`, `qwen`, `amp`, `pi` variants, `cursor`, `copilot`, and any TOML agent inheriting `DoneStrategy(type="api")` from `src/dgov/agents.py:24`) | Agent process never emits `dgov worker complete` / done signal and may never produce a commit when the provider stalls mid-response. | `_is_done` relies on signal files, liveness, and the `done_strategy=api` fallback after 60 s of stable output (`src/dgov/done.py:310`). | `wait_worker_pane` keeps polling until the 600 s default timeout and then raises `PaneTimeoutError` (`src/dgov/waiter.py:232` and `src/dgov/waiter.py:293`), setting the pane state to `timed_out` while the underlying CLI may keep running silently. |
| Stable-done agents (`cline` and `crush`, configured at `src/dgov/agents.py:130` with `DoneStrategy(type="stable", stable_seconds=30)`) | CLI keeps streaming logs or progress so output never stabilizes, meaning `_is_done` never writes a done signal. | `_is_done` only marks the pane done once the output tail has not changed for `stable_seconds` (`src/dgov/done.py:346`); transient logging resets `stable_since`. | As long as the stream keeps moving, the `waiter` loop never returns and the platter sleeps until `timeout=600` triggers `PaneTimeoutError`; only repeated identical output eventually trips the circuit breaker (`src/dgov/done.py:376`), but that only happens after many polls. |

## 2. Binary-present vs service-running gap

* `check_agent_cli` only verifies the leading token of `prompt_command` is on `PATH` (`src/dgov/preflight.py:60`) and never confirms the CLI can contact its backend. All built-ins inherit the null `health_check` field from `AgentDef` (`src/dgov/agents.py:41`), so a broken network, revoked API key, or misconfigured proxy is invisible until a pane is already running.
* `done_strategy=api` forces a human/agent to emit a done signal after the work, so any interruption before that point keeps the pane running until the 600 s timeout. The risk is independent of the binary check and affects every agent that does not override `health_check`.
* The only service-level probe in this install is the custom `pi` entry in `~/.dgov/agents.toml:9`. Because the probe targets `localhost:8080`, it never sees the actual River service and keeps failing the preflight check instead of letting `health_fix` boot the right backend.

## 3. Race conditions in the health-check → launch → wait pipeline

1. **`max_concurrent` is not atomic.** Both `check_agent_concurrency` (`src/dgov/preflight.py:323`) and the in-flight guard in `create_worker_pane` (`src/dgov/lifecycle.py:452`) simply read `_count_active_agent_workers`. The `state.json` record for a pane is written only after `add_pane` (`src/dgov/lifecycle.py:505`), so there is a multi-second window between the check and the write where a second invocation can start, resulting in multiple panes exceeding `max_concurrent` before the guard sees them.
2. **Health-check TOCTOU between preflight and launch.** `run_preflight` runs `check_agent_health` with a 10 s timeout (`src/dgov/preflight.py:363`), but `create_worker_pane` reruns the same command without any timeout (`src/dgov/lifecycle.py:441`). The system therefore assumes the health check result is still valid, but the agent service can fail between the two calls; when that happens, `create_worker_pane` hangs forever because the second probe never times out, and the user never gets a meaningful error.
3. **Done detection depends on stale pane metadata.** `_poll_once` reuses the `pane_record` captured before the loop (`src/dgov/waiter.py:252`), so metadata like the `pane_id` or `project_root` is not refreshed while the pane is still running. If the pane is closed and resumed, or if auto-retry kicks in (`src/dgov/waiter.py:274`), the next loop iteration still queries the old `pane_record`, delaying detection of the new slug and temporarily leaving the wrong pane marked as pending.

## 4. What a proper per-agent health probe would look like

1. **Phase 1 – binary verification.** Keep `check_agent_cli` but also verify version compatibility (e.g., run `cli --version` and parse output) so upgrades that change the CLI token fail fast.
2. **Phase 2 – service handshake.** Run `agent_def.health_check` (when configured) but wrap it with a short timeout plus a retry/backoff and compare the response body or exit code to expected values. Record latency and surfaced message so the user can tell whether it is a network failure, authentication failure, or not found.
3. **Phase 3 – integration-level smoke test.** Launch the CLI with a harmless prompt or `--health` flag that exercises the provider path (for `pi`, hit the real River host rather than `localhost`). If the probe fails, capture the output and run `health_fix` where available.
4. **Phase 4 – gating and state.** The health probe should flag both the CLI binary and the remote endpoint together: only pass when the service handshake succeeded and the provider returned a usable token. Record the probe result in the pane metadata so the wait loop can differentiate between a CLI startup failure and a backend outage.

## 5. Is the 10-minute default timeout appropriate?

`wait_worker_pane` defaults to `timeout=600` seconds and uses that to raise `PaneTimeoutError` when the loop has polled long enough (`src/dgov/waiter.py:232` and `src/dgov/waiter.py:293`). This is brittle for both sides of the spectrum:

* Long-running agents (large models, multi-step searches) need more time than 10 min; forcing them to restart just pushes the problem down the stack.
* Short-lived agents that fail fast leave the user waiting up to 10 min for the timeout. When a health check already told us the backend was reachable, the wait loop should be adaptive: reuse the agent’s `done_strategy`/`stable_seconds`, tie the timeout to the expected commit cadence, or respect a per-agent `max_duration` field instead of a hard 600 s.

An adaptive timeout could be the maximum of the current `stable_seconds`, a short API-specific bound (say 3× the average latency seen during the last success), and an upper cap (e.g., 30 m). That way, interactive agents still get a reasonable cutoff, and long-running jobs don’t falsely trigger `PaneTimeoutError`.
